## 4. Data schema

Every entry in `data.js`/`data2.js`/`data3.js`/`data4.js` is a plain JS object:

```js
{ model: string, attribution: string|null, subject: string, text: string }
```

- **`model`** — the tool/platform, as stated by the source. `"Unspecified"` (+ parenthetical context) if unknown.
- **`attribution`** — who created/shared it (`"Vigo Zhao (@VigoCreativeAI)"`, `"D the Designer"`, etc.), or `null`/omitted if genuinely unknown. **Do not guess this.** Only set it when the source explicitly states it, or when content is unmistakably D's own (e.g. it uses her established fictional-universe terminology like "Orphan Sky," "CoModularism," "Erebus," "AWC," "ReDep" — these are safe to attribute to "D the Designer" even without an explicit statement, because they're her own established IP).
- **`subject`** — a short human-readable label. Hand-write these for anything you add one at a time; for bulk imports, auto-derive by trimming to ~70–90 chars at a word boundary.
- **`text`** — the prompt, verbatim, **except** for specific redacted tokens (see §5).

At build time (inside `build.js`/`build_html.js`), two more fields get computed and must never be stored in the data files themselves:
- **`id`** — the entry's 1-indexed position in `data1.concat(data2).concat(data3).concat(data4)`. This is the permanent "Archive #" shown everywhere. **New entries always go at the end of the last file** (currently `data3.js` — `data4.js` is a closed, one-time bulk import, don't append to it; start a `data5.js` if you want a new closed batch, or just keep growing `data3.js`). Never insert in the middle — that would renumber everyone after the insertion point.
- **`category`** — computed by `classify()` in `categorize.js`, not stored, recomputed on every build.

## 5. Redaction policy — read this before adding anything with a real name in it

**Default: include everything verbatim.** Only redact when a prompt does one of these two things:

1. **Names a real, identifiable, living public figure** in a way that reads as generating a realistic fabricated scenario about them specifically (a staged "candid" photo, implied real private conduct). Replace the name with `[public figure redacted]`.
2. **Names a specific copyrighted character, vehicle, franchise-specific location, or exact-title-as-style-clone-target** as the literal subject (not a genre/aesthetic reference). Replace with `[redacted IP]`.

**Do NOT redact:**
- Real names used purely as a likeness/style anchor for clearly fictional character art ("actor X as a space wizard," a historical figure's portrait). This is completely normal creative-prompt practice.
- Genre/aesthetic references to real films, artists, or franchises ("blade runner cyberpunk aesthetic," "in the style of Caravaggio," two or more classic film titles cited together as mood references).
- A franchise name used **only as a negative/avoid instruction** ("no Star Wars style clutter," "Avoid: ... Transformers, Star Wars droids"). Telling the model what NOT to look like is the opposite of an infringement request.

**Every redaction gets logged.** There's a hardcoded `redactions` array near the top of `build.js` — each entry has `{where, what, replacement, why}`. Add a new object there for every redaction you make. This ships as its own page/section in the Word doc, right after the title page, before the prompts begin.

A subtlety that bit us once: the redaction marker strings `[redacted IP]` and `[public figure redacted]` are themselves lowercase-bracketed, which means a naive "is this a `[bracket-placeholder]` template" check will misfire on them. See §6 for how this was fixed — don't reintroduce that bug if you touch the classifier.

## 6. The category classifier (`categorize.js`) — known gotchas

`categorize.js` exports `{ classify(entry), CATEGORIES }`. `classify()` runs a fixed, ordered list of keyword/regex checks — **first match wins**, so ordering is load-bearing. Current order (roughly, check the file for the literal sequence): JSON/system-prompt detection → GPT-workflow detection → graphic-design-technique detection → **video/camera-motion detection → placeholder-template detection** (this pair had to be reordered, see below) → reference-list detection → real-person-likeness → retrofuturism → biomechanical → fantasy → fashion → anime → horror → posters → space → product → general catch-all.

**Three real bugs already found and fixed here — don't reintroduce them:**

1. **Substring collisions.** A keyword like `"mage"` will match inside the word "i**mage**" via plain `.includes()`. Fixed with `\bmage\b` word-boundary regex. If you add new short keywords, sanity-check them against the word "image" and similar common words before shipping — this quietly mis-categorized dozens of entries into "Fantasy" until caught.
2. **Redaction markers look like templates.** `[redacted IP]` and `[public figure redacted]` are lowercase-bracketed strings, so a case-insensitive `\[[A-Za-z]...\]` placeholder-detector matches them as if they were user-supplied `[SUBJECT]`-style template placeholders. Fixed by stripping those two specific strings out of the text before running the placeholder regex. If you broaden the placeholder regex again, re-strip these first.
3. **Priority ordering matters more than it looks.** A prompt using a named video model (Kling, Seedance, Runway) but that also happens to contain one incidental `[bracket]` used to mean something else (e.g. `[IMAGE REFERENCE]`) should be classified as Video, not Templates — but if the placeholder check runs first, it wins wrongly. Video-motion detection was moved to run *before* the placeholder check for this reason.

**Whenever you change a rule, re-run classification across the whole corpus and diff against the previous result before shipping** — don't just eyeball the one new entry. Pattern used throughout this project:
```js
const data = data1.concat(data2).concat(data3).concat(data4);
const { classify, CATEGORIES } = require('./categorize.js');
const counts = {};
data.forEach(d => { const c = classify(d); counts[c] = (counts[c]||0)+1; });
CATEGORIES.forEach(c => console.log(c, '->', counts[c]||0));
```
Compare category counts before/after your change; investigate anything that moved unexpectedly.

## 7. How the big Midjourney import (`data4.js`) was built

D provided two raw scraper-export markdown files (her actual Midjourney job history, ~2,583 job entries including retries). The process:
1. Parsed `## Entry N` blocks, extracted the `Prompt:` field.
2. Deduplicated by stripping image-reference URLs and MJ flags (`--v`, `--profile`, etc.) and grouping identical remaining text — 2,583 raw entries collapsed to 809 unique prompts.
3. Per the person's explicit instruction, filtered out anything under 15 words of actual descriptive text (drops low-effort one-liners) — 721 remained.
4. For groups with multiple retries, kept the longest/fullest variant and appended a note: `"(Generated N times in the archive with varying seeds/flags — this entry shows the fullest version of the prompt.)"`.
5. Scanned the kept set for real-person/copyrighted-franchise redaction concerns before import (see §5) — found and redacted 3 near-duplicate entries.

This was a one-time bulk operation — **don't re-run it or merge new raw archive exports into `data4.js`**; if D provides more raw archive files later, treat it as a fresh scope decision (ask her about dedup/filter thresholds again, same as the first time) and put it in a new file.

## 9. The Builder tool (`AI_Prompt_Compendium_Builder.html`) — feature inventory

This file is hand-edited directly, not generated. It's a single self-contained HTML file (embedded CSS/JS, no build step, no external dependencies except the inline base64-embedded font — see §10). Features, so you know what's already built before "helpfully" re-adding something:

- Add/Edit/Delete single prompts via a modal form, with an **Auto-suggest** category button (a simplified JS port of `categorize.js`'s logic, kept in sync by hand — if you change the real classifier meaningfully, update this port too).
- **Bulk Add**: paste a wall of text separated by blank lines or `---`, auto-splits, auto-detects attribution (`Name (@handle)` pattern), auto-suggests category per block, shows a review screen with per-item checkboxes before committing.
- **Demo/starter data**: 9 example entries seeded automatically on a person's very first visit (detected via a separate `localStorage` flag, not just "entries array is empty," so intentionally clearing your data doesn't re-trigger the demo). A "Clear examples & start fresh" button removes only `isDemo: true` entries.
- **Two category-taxonomy options**: a neutral default set, plus a "Try example categories" button that swaps in the more elaborate 17-category system used in D's own compendium.
- **Export/Import** as JSON (replace or merge), with an **export-nudge banner** that appears if real (non-demo) content exists and it's been 7+ days (or never) since the last export.
- **Per-entry reference-image drop zone**: drag an image onto a small square next to Copy, stored locally via IndexedDB keyed by Archive #, click to view full-size in a new tab, hover+click × to remove. **The Copy button never includes the image — verified this explicitly with a functional test**, not just by reading the code. If you touch either the copy logic or the image logic, re-verify they're still independent.
- **Print/Save PDF** via the browser's native print dialog (a `@media print` stylesheet hides the sidebar/controls).

The populated `compendium.html` (from `build_html.js`) has the read-only subset of this: search, model/category filters, sidebar table-of-contents (collapsible per category), Copy buttons, and the same per-entry image-drop feature. It does NOT have Add/Edit/Bulk-Add — it's meant to be regenerated from the data files, not hand-edited in the browser.

## 10. Visual identity — do not casually restyle this

- **Palette**: near-black background (`#030a03`), bright phosphor green accent (`#33ff33`), everything else derived from CSS custom properties (`--bg`, `--panel`, `--panel-border`, `--ink`, `--ink-dim`, `--accent`, `--accent-dim`) — **always use the variables, never hardcode a hex color**, or a future palette change will miss it (this happened once already with a few `#0d0f0d` hardcoded button-hover colors from before the green-theme switch; caught and fixed, don't reintroduce).
- **Scanline + vignette overlay** on `body` via CSS `repeating-linear-gradient` + a `body::after` radial-gradient vignette. A subtle `text-shadow` phosphor glow is applied to headers/labels (`.eyebrow, .cat-heading h2, .card-n, h1`).
- **Font**: real OCR-A (not a lookalike). Specifically [John Sauter's Public Domain OCR-A reproduction](https://sourceforge.net/projects/ocr-a-font/), ANSI X3.17-1977 conformant. **Embedded inline as base64** inside a `@font-face` rule (not linked externally) so both files remain fully self-contained/offline-capable. Every text element routes through `var(--mono)`, `var(--ocr)`, or `var(--sans)` — all three currently point to the same OCR-A font stack. If the person ever wants body text back in a more readable font while keeping OCR-A for headers/labels only, that's a one-variable change (`--mono`/`--sans` back to something like IBM Plex, leave `--ocr` alone) — don't hand-edit every individual CSS rule.
- **Footer credit**: `compendium.html` says "Curated by D the Designer"; the Builder says "Tool by D the Designer" (different framing on purpose — one's her content, one's a tool for other people's content). Both link to `https://x.com/D_the_Designer`.

## 11. GitHub push workflow

There's no CI/automation — pushing is manual, done via `git` from a local clone with a Personal Access Token. In this conversation, the person provided a classic PAT (scope: `repo`) directly in chat when asked; it was used to push and then the working directory's git remote was removed immediately after each push (`git remote remove origin`) so the token wasn't left sitting in a config file. **You'll need your own auth path** — ask the person for a token (recommend a fresh one scoped narrowly, and recommend they revoke/rotate it once you're done, since anything typed in a chat transcript should be treated as semi-exposed) or use whatever GitHub integration you have natively.

Both repos get the same three files pushed on every update: `compendium.html`, `index.html` (the Builder), and this handoff/README/spec if changed. Typical push pattern used throughout:
```bash
cd <local clone or working copy>
git add -A
git commit -m "<description>"
git remote add origin "https://x-access-token:$TOKEN@github.com/D-the-Designer/<repo>.git"
git push origin main
git remote remove origin
```
Repeat for both `AITools` and `prompt-compendium`.

## 12. Verification pattern — do this on every change, not just big ones

Before telling the person something is done, this project's whole working pattern has been:
1. Parse/add data → check it actually `require()`s without a syntax error.
2. Run `classify()` on the new entry (and re-run the full-corpus distribution check per §6 if you touched `categorize.js`).
3. Rebuild both `.docx` and `.html`.
4. Diff `id` and `category` between the two outputs, entry-by-entry — should be **zero mismatches**, always. This is the single most important sanity check in the whole project, because both files are generated from the same data by two independent scripts and any divergence means something is broken in one of them.
5. For anything UI/interactive (the Builder's JS), actually simulate the interaction (jsdom + a fake IndexedDB where needed) rather than just reading the code and assuming it works. This caught real bugs more than once (a broken categorize.js edit, a `URL.createObjectURL` environment gap, a font-encoding round-trip that needed verifying byte-for-byte).
